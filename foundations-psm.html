The problem-solving methodology

Every digital solution in this course is built with the same four-stage approach — analysis, design, development and evaluation. Master the methodology once and it underpins every outcome, from a data visualisation to a secure network response.

1 · Analysis

Analysis works out what the solution must do before any building begins. The output of analysis is a clear, agreed statement of the solution requirements, constraints and scope — sometimes formalised as a Software Requirements Specification (SRS).

Solution requirements

  • Functional requirements — what the solution must do — described with verbs (e.g. “calculate the weekly total”). KK08 Requirements, constraints & scopeKK04 Requirements, constraints & scopeKK05 Solution specifications
  • Non-functional requirements — how well the solution must perform — described with adjectives (usability, reliability, robustness, maintainability, portability, performance, ease of use). KK08 Requirements, constraints & scopeKK04 Requirements, constraints & scope
  • Constraints — limitations the solution must work within — economic, legal, technical, social and usability factors. KK08 Requirements, constraints & scopeKK04 Requirements, constraints & scope
  • Scope / design brief — the boundaries of the solution — what is in, and explicitly what is out. KK08 Requirements, constraints & scope
A good SRS is unambiguous, easy to understand, breaks the problem down, and tells developers what must be achieved (and what is out of scope). It separates logical needs (the vague “what”) from physical detail (the precise “how”).

Discovering requirements

Requirements are gathered from users and stakeholders using a range of techniques:

  • Interviews — one-to-one questioning that gathers rich, detailed qualitative needs. KK06 Collecting user needs
  • Surveys & questionnaires — structured questions distributed widely for quantitative data at scale. KK06 Collecting user needs
  • Observation & on-site visits — watching users in their real context to uncover needs they may not state. KK06 Collecting user needs

Analysis tools

  • Context diagram — shows the solution as a single process and the external entities it exchanges data with.
  • Use case diagram — shows the actions (use cases) different users can perform with the system.
  • Data dictionary — defines every data element: its name, type, size and validation.
  • User context diagram (UCD) — places the user at the centre and maps their interactions and environment.

2 · Design

Design decides how the solution will look and work, and sets the evaluation criteria that will later judge success. Designs are represented with design tools and reviewed before development.

Logical design tools (the “what”)
Capture meaning and flow without committing to appearance: context diagrams, data dictionaries, data flow diagrams (DFD), hierarchy/structure charts, decision trees and pseudocode.
KK07 Design tools
Physical design tools (the “how”)
Capture the concrete look and structure: mock-ups and layout diagrams, storyboards, flowcharts, structure charts, and entity-relationship diagrams (ERDs).
KK09 Design tools for solutionsKK07 Design tools

Common design tools

  • IPO chart — lists the Inputs, the Process and the Outputs of a solution. KK09 Design tools for solutions
  • Mock-up / layout diagram — a draft of the on-screen appearance and placement of elements. KK09 Design tools for solutionsKK07 Design tools
  • Flowchart & pseudocode — step-by-step logic of an algorithm, language-independent. KK07 Design tools
  • Storyboard & sitemap — the sequence of screens and the navigation between them. KK07 Design tools
  • Entity-relationship diagram (ERD) — models data entities and their relationships; drawn in Chen’s notation or crow’s-foot notation. KK10 Relational database structure
  • Data flow diagram (DFD) — shows how data moves between processes, stores and external entities. KK09 Design tools for solutions

Evaluation criteria & factors influencing design

  • Evaluation criteria — measurable statements of efficiency (cost/time) and effectiveness, written now and used in Stage 4. KK14 Evaluation criteria
  • Factors influencing design — economic, legal, technical, social and usability factors (affordance, interoperability, marketability, security).

3 · Development

Development turns the design into a working solution through manipulation (coding/building), then makes it trustworthy through validation, testing and documentation.

Manipulation & documentation

  • Manipulation (coding) — building the solution — entering formulas, writing code, structuring data. KK08 Features of a languageKK12 Manipulating & validating data
  • Internal documentation — meaningful names and comments inside the solution that explain its data and logic and support maintenance. KK10 Internal documentationKK09 Naming conventions

Validation — is the input reasonable?

  • Existence check — confirms required data has been entered. KK12 Manipulating & validating dataKK11 Validation techniques
  • Type check — confirms data is the expected data type. KK12 Manipulating & validating dataKK11 Validation techniques
  • Range check — confirms a value sits within an allowed minimum and maximum. KK12 Manipulating & validating dataKK11 Validation techniques

Testing — does it work?

  • Test table — lists test cases with input, expected output and actual output. KK12 Debugging & testing
  • Trace table & desk check — manually step through logic, tracking variable values to find errors.
  • Test data — valid, invalid and boundary values chosen to exercise the solution. KK12 Debugging & testing
Validation ≠ testing. Validation checks that input data is reasonable; testing checks that the solution’s logic produces the correct output.

4 · Evaluation

Evaluation judges how well the finished solution met its requirements. It happens after the solution has been in use, against the criteria written during design, following an evaluation strategy.

Strategy, criteria & measures

  • Evaluation strategy — the plan for how and when the solution will be evaluated — what data to collect and over what timeframe. KK14 Evaluation criteria
  • Efficiency — how economically the solution uses resources — speed, labour and cost. KK14 Evaluation criteria
  • Effectiveness — how well the solution achieves its purpose — quality, accuracy, reliability, attractiveness/readability, accessibility, portability, security, compatibility and expandability/flexibility. KK14 Evaluation criteria

Evaluation methods

  • Interviews & surveys — collect real user feedback after the solution is in use to measure effectiveness. KK06 Collecting user needsKK10 Testing & evaluating
  • Comparing against criteria — measure the solution against each efficiency and effectiveness criterion set in design.
